Introduction  
L’objectif de ce projet est de mettre en application les concepts orientés objet et les 
notions de programmation Java vus en cours. Il consiste en la réalisation d’une application 
client-serveur de gestion des livraisons pour une entreprise de livraison de colis. La partie 
serveur centralise la gestion des livraisons, tandis que la partie client implémente les 
opérations du livreur. 
Afin de vous concentrer sur l’objectif principal du projet, vous disposerez du noyau de 
la partie serveur, qui gère les communications avec le client, ainsi que de la partie 
client complète. Votre travail consistera donc à compléter la partie serveur, que vous 
réaliserez en deux travaux pratiques : 
1. Travail pratique 1 (TP1) : compléter la partie serveur en implémentant le module 
de gestion des livraisons en mettant en pratique les concepts orientés objet. 
L’utilisation des applications client et serveur se fera en mode console. 
2. Travail pratique 2 (TP2) : implémenter l’interface graphique présentant les 
activités de livraison. 
 
 
Ce travail vous permettra également d’avoir un aperçu de : 
 La programmation réseau avec les sockets, où un serveur communique avec des 
clients ; 
 La programmation multithread en Java.

Gestion des livraisons  
Pour son activité de livraison de colis, l’entreprise collabore avec des livreurs utilisant 
trois moyens de déplacement : vélo, voiture et camion. Chaque livreur possède un 
numéro qui l’identifie (id) et son nom. Il effectue un certain nombre de livraisons selon 
le moyen de déplacement dont il dispose : il s’agit de sa capacité de livraison. 
Une livraison possède un identifiant numérique unique (id) et peut être régulière ou 
urgente. Les livraisons sont regroupées en lots numérotés séquentiellement. Ainsi, les 
livraisons du lot 1 sont attribuées aux livreurs, puis celles du lot 2, et ainsi de suite. En 
cas d’échec d’une livraison, celle-ci peut être tentée de nouveau jusqu’à trois fois. 
Une livraison peut être dans l’un des quatre états suivants, il s’agit du statut de la 
livraison :  
- 
En attente : indique que la livraison n’a pas encore été attribuée à un livreur  
- 
En cours : indique que la livraison a été attribuée à un livreur 
- 
Livrée : indique que la livraison a été effectuée avec succès 
- 
Échouée : indique que la livraison a échouée (par exemples pour mauvaise 
adresse, destinataire absent, etc.)  
Au départ de l’application, le serveur dispose d’un nombre de livraisons stockées dans 
l’ordre selon lequel elles doivent être réalisées. Il s’agit de la liste des livraisons à 
effectuer. Ainsi, lorsqu’un livreur se connecte et demande de recevoir des livraisons, 
le serveur lui envoie un nombre de livraisons selon sa capacité. Sachant que la capacité 
de livraison dépend du moyen de transport utilisé par le livreur : vélo, voiture ou 
camion. 
Note : lorsqu’une une livraison est attribuée pour la première fois à un livreur, il s’agit 
là de la première tentative.  
Une fois que le livreur reçoit ces livraisons, il communique par la suite, avec le serveur 
pour lui indiquer pour chaque Livraison si elle a été effectuée avec succès ou si elle a 
échouée. 
Les règles suivantes régissent l’activité de livraison : 
- 
Le serveur envoie à un livreur une liste de livraisons, puis les retire de la liste 
des livraisons à effectuer. 
- 
Le client indique que la livraison a été effectuée avec succès. Le serveur va

Présentation du code fourni  
Pour la réalisation de ce travail, vous disposez d’un code de démarrage composé de 2 
applications :  
- 
Le serveur (projet miniServeur). Les tâches principales du serveur sont la gestion 
des livraisons et l’interaction avec les livreurs.  
- 
Le client (projet miniClient). Cette application permet aux livreurs de se

LE SERVEUR 
Au lancement, le serveur commence par lire le fichier livreurs.txt contenant les livreurs 
enregistrés. Ensuite, il entre dans une phase d’initialisation qui consiste principalement 
à lancer un thread qui se met à l’attente de connexions de la part des clients et un 
thread qui réceptionne les commandes envoyées par les clients connectés.  
Le rôle du premier thread est de d’enregistrer les clients qui se connectent.  
Le rôle du deuxième thread fait la réception des commandes envoyées par les clients 
connectés et les traite. Le traitement d’une commande revient principalement à faire 
la gestion des livraisons : attribuer des livraisons au livreurs, collecter les livraisons 
effectuées, traiter les échecs des livraisons, etc. Le traitement des commandes est 
délégué à la méthode nommée traiter(Evenement evenement) de la classe principale 
de gestion des livraisons : GestionnaireLivraisons

Partie 2 : Traitement des commandes 
Dans la partie 2 du TP, vous allez implémenter le traitement des commandes reçues 
par le serveur. En effet, les livreurs envoient des commandes au serveur par le biais de 
l’application client (MiniClient). La communication entre les 2 applications utilise le 
mécanisme de sockets, implémenté dans la librairie standard de Java.  
Les événements 
L’élément d’interaction entre 2 entités qui communiquent, est l’événement. Dans 
notre contexte, l’événement correspond à la réception d’une commande par le serveur 
ou la réception d’une réponse par le client. Pour représenter un tel événement, on a 
défini la classe Evenement (package com.atoudeft.commun.evenement) :  
public class Evenement { 
    private final Object source; 
    private final String type, argument; 
 
    /** 
     * Construit un événement. 
      * @param source   La source de l'événement 
     * @param type     Le type de l'événement 
     * @param argument Chaîne de caractères contenant les arguments de l'événement 
     */ 
    public Evenement(Object source, String type, String argument) { 
        this.source = source; 
        this.type = type;

Arguments args = new Arguments(evenement); 
String unArgument = args.extraireArgumentSuivant(); 
String leResteDesArguments = args.lire();

public void traiter(Evenement evenement) { 
    Object source = evenement.getSource(); 
 
    if (source instanceof Connexion) { 
        Connexion cnx = (Connexion) source; 
         
        String reponse = ""; 
        switch (evenement.getType().toUpperCase()) { 
            case "EXIT": // Le client se déconnecte. 
                reponse = this.traiterEXIT(evenement); 
                break; 
            case "REGISTER": // Le client s'enregistre comme nouveau livreur. 
                reponse = this.traiterREGISTER(evenement); 
                break; 
            case "ID": // Le client s'identifie. 
                reponse = this.traiterID(evenement); 
                break; 
            case "GET": //  Le client a demandé des livraisons à effectuer. 
                reponse = this.traiterGET(evenement); 
                break; 
            case "DELIVERED": // Le client informe qu'une livraison a été effectuée. 
                reponse = this.traiterDELIVERED(evenement); 
                break; 
            case "FAILED": // Le client informe qu'une livraison n'a pas pu être effectuée. 
                reponse = this.traiterFAILED(evenement); 
                break; 
            case "INCOME": 
                reponse = this.traiterINCOME(evenement); 
                break; 
            case "INFO": 
                reponse = this.traiterINFO(evenement); 
                break; 
            case "SEND": 
                reponse = this.traiterSEND(evenement); 
                break; 
            default: // La commande envoyée par le client n'a pas été reconnue. 
                this.traiterCOMMAND_ERROR(); 
        } 
        cnx.envoyer(reponse); 
    } 
}

Partie III : messagerie entre livreurs 
Une autre fonctionnalité de l’application client-serveur pour la gestion des livraisons, 
est la messagerie entre les livreurs. Ainsi, un livreur a la possibilité d’envoyer un 
message à un livreur en particulier ou à l’ensemble des livreurs authentifiés (diffusion) 
grâce à la commande SEND :  
- 
SEND <id livreur> <texte du message> : envoie un message à un livreur en 
particulier 
- 
SEND * <texte du message> : envoie un message à tous les autres livreurs. Noter 
que * c’est pour indiquer que c’est une diffusion 
Lorsqu’un livreur envoie un message, c’est le serveur qui le récupère pour le 
transmettre à un livreur en particulier ou à l’ensemble des autres livreurs authentifiés. 
Notez que le message reçu au niveau du Serveur contient un identifiant unique 
(identifiant_msg) et un contenu.   
Après avoir transmis le message, le serveur répond au livreur expéditeur par la 
commande ACK <identifiant_msg>  
Cependant, à cause de l’instabilité de la communication, des messages peuvent être 
perdus et ne jamais être reçus par le serveur. Pour pallier ce problème, le client envoie 
de manière répéter le même message (tous les 500 millisecondes) jusqu’à la réception 
de la commande ACK de la part du Serveur, prouvant que le message a bien été reçu.  
Par conséquent, le serveur peut recevoir plusieurs fois un même message d’un livreur, 
la première fois, le message est traité (c’est-à-dire transmis au(x) destinataire(s)), pour 
les autres fois, le message est simplement ignoré.  
Question 3.1 : traitement de la commande SEND 
Pour le traitement de la commande SEND, compléter la méthode traiterSEND(Evenement 
evenement). Parmi les opérations de cette méthode : 
- 
S’assurer que le livreur expéditeur s’est bien authentifié 
- 
Détermine si ce même message (selon son identifiant) a déjà été reçu et traité, 
dans ce cas, le message est ignoré  
- 
Si le message est reçu pour la première fois, il est transmis au destinataire ou 
diffusé à tous les autres livreurs authentifiés   
- 
Retourner la commande ACK <identifiant_msg> à l’expéditeur 
Attention : il faut traiter les cas d’erreur comme destinataire non authentifié, 
paramètres non valides, etc.
